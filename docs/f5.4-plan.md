# Feature: F5.4 — Scaled rounding-mode `nppiConvert` variants

## Preconditions (verified before Phase 1)

- `NPP_TYPE_TOKENS` (`npp-codegen/src/classify.rs:160`) contains all F5.4
  tokens (`32u`, `32s`, `16u`, `16s`, `8u`, `8s`, `32f`) — no pair will
  silently drop.
- `NppPixelType` impls exist for all F5.4 types in `npp/src/image.rs:44-79`
  (u8, i8, u16, i16, u32, i32, f32, f64).
- `RoundMode` re-export path: `use npp_rs::imageops::RoundMode` (matches
  existing golden test imports).
- Before Phase 3 Step 1: confirm `npp-codegen/Cargo.toml` has no explicit
  `[[example]]` block (else register `gen_convert_round_scaled_impls`).

## Execution split (CRITICAL for the driver)

- **Pipeline-completable on any host (no GPU needed):** Phase 0, Phase 1, all
  pure-logic tests (codegen guard tests, unit tests).
- **Requires CUDA host + `npp-sys` built + real GPU (mark
  `pending-manual-GPU`, NOT done):** Phase 2 Step 1 (D-API: verify
  `nScaleFactor` semantics from NPP header), Phase 3 Step 3 (symbol-gate
  test), Phase 4 Steps 1/3/4 (shape-check test + per-mode golden + chained
  golden). These phases cannot complete without a GPU; the plan's pipeline
  runner must not falsely report success.

## Design summary (read once before execution)

**Goal:** NPP exposes `nppiConvert_<src>_<dst>_<round>C1RSfs` (and `_Ctx`
variants) — single-channel conversions with a rounding mode AND a scale factor
(`Sfs` suffix convention, expected to be a power-of-two exponent). Ship a
`ConvertRoundedScaled<Dst>` trait following the F5.3 `ConvertRounded<Dst>`
codegen pattern.

**Breadth contract:** npp-rs binds every NPP symbol family exposed by NPP.
"Name the user" is not the bar — all 17 `(src,dst)` pairs are included. The
executor trims only when a symbol fails the existence gate.

**No multi-channel (`C3RSfs`/`C4RSfs`):** Verified zero matches in
`bindings.rs`. All 34 scaled convert symbols are `C1RSfs` (single-channel)
only. The fixture must not list multi-channel symbols.

**Latent-bug fixes folded into F5.4 as Phase 0 (separate first commit):**
1. Replace broken `find` shell-out in `npp-codegen/src/main.rs` survey binary
   with `gen_impls::find_bindings_rs()` (the correct resolver already used by
   guard tests).
2. Add `example_name` field to `FamilyDescriptor` and set on all 5 existing
   descriptors to fix wrong regenerate-command in generated file headers.

**Dispatch selector:** Add `dual_type_round_scaled: bool` (follows existing
`dual_type`/`dual_type_round` precedent in `classify.rs` and `gen_impls.rs`).

**Channel rejection:** Explicit `if self.channels() != 1 { return Err(...) }`
guard in the macro, not reliance on the generic `_ =>` match arm.

**`nScaleFactor` semantics (D-API gate):** The `nScaleFactor` parameter
appears in NPP as `int nScaleFactor`. The NPP `Sfs` naming convention
suggests it is a power-of-two exponent (where the actual scale factor is
`2^nScaleFactor`), but this must be **verified against the NPP header or
online docs** before Phase 2 (trait signature + macro). Full verification
procedure with decision tree is described in Phase 2 Step 1.

**Golden test matrix (F5.3 parity):** One per-mode golden test with
CPU-generated input + pinned `EXPECTED`, plus one chained `_Ctx` golden test.
Chained test is low-risk (already single-channel per F5.3 precedent).

use npp_rs::image::CudaImage;
use npp_rs::imageops::{ConvertRoundedScaled, RoundMode};
use npp_rs::stream::stream_context_for;
use std::convert::TryFrom;

const SRC_W: u32 = 8;
const SRC_H: u32 = 4;

#[test]
fn test_convert_rounded_scaled_shape() {
    let ctx = stream_context_for(0).expect("CUDA device init");
    let src_data: Vec<u8> = (0..SRC_W * SRC_H * 1).map(|i| (i * 37) as u8).collect();
    let src = CudaImage::from_host(ctx.clone(), 1, SRC_W, SRC_H, &src_data)
        .expect("src allocation");
    let mut dst = CudaImage::<f32>::new(ctx.clone(), 1, SRC_W, SRC_H)
        .expect("dst allocation");
    src.convert_rounded_scaled(&mut dst, RoundMode::Nearest, 0)
        .expect("convert_rounded_scaled");
    assert_eq!(dst.width(), SRC_W);
    assert_eq!(dst.height(), SRC_H);
    assert_eq!(dst.channels(), 1);
}
```

use npp_rs::test_helpers::assert_golden;
use std::convert::TryFrom;

const W: u32 = 8;
const H: u32 = 4;

fn make_input() -> Vec<u8> {
    // Procedural gradient input
    (0..W * H * 1).map(|i| {
        let x = i % W;
        let y = i / W;
        ((x * 31 + y * 17) & 0xFF) as u8
    }).collect()
}

// GPU-captured bytes — empty until pinned on a GPU host.
const EXPECTED: &[f32] = &[];

#[test]
fn test_golden_convert_round_scaled() {
    let ctx = stream_context_for(0).expect("CUDA device init");
    let src = CudaImage::from_host(ctx.clone(), 1, W, H, &make_input())
        .expect("src allocation");
    let mut dst = CudaImage::<f32>::new(ctx.clone(), 1, W, H)
        .expect("dst allocation");
    src.convert_rounded_scaled(&mut dst, RoundMode::Nearest, 0)
        .expect("convert_rounded_scaled");
    let output: Vec<f32> = Vec::try_from(&dst).expect("read-back");
    assert_golden(&output, EXPECTED, "convert_rounded_scaled u8->f32 Nearest scale=0");
}
```

The input design (gradient formula) and `scale_factor` value may need tuning
based on the D-API finding (Step 2.1). The `assert_golden` function handles
the "not yet pinned" panic.

Choose the pixel type pair based on the first fixture entry where `src` and
`dst` are a natural golden-test pair: `u8 -> f32` with `scale_factor = 0`
(same as identity). If `nScaleFactor` is a power-of-two exponent, `0` means
`2^0 = 1`, i.e., identity scale — no quantization artefacts to debug.
